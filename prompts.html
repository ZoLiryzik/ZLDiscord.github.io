📋 Пошаговые промпты для OpenCode

✅ Каждый промпт отправлять строго по очереди. Проверить работу → следующий.

1. Фундамент и архитектура

Шаг 1 
Создай основу проекта на Unity (C#) или Godot (C#) — выбери Godot 4 C# для кроссплатформенности.

Реализуй центральные интерфейсы и DI-контейнер (ручной Service Locator или через конструкторы). Структура папок: Core, Gameplay, Network, UI, Data.

Сделай интерфейсы:
- IInputProvider (получение ввода)
- IGameState (текущее состояние игры)
- IModLoader (загрузка/перезагрузка .txt конфигов)
- INetworkService (абстракция над LiteNetLib)
- IAssetProvider (загрузка спрайтов/звуков)

Создай пустые заглушки для этих сервисов.

Цель: получить компилируемый проект с внедрёнными зависимостями и готовой архитектурой для следующих шагов.

2. Парсер .txt и горячая перезагрузка

Шаг 2 
На основе интерфейса IModLoader реализуй:
- Чтение .txt файлов из папки mods/ в Dictionary (ключ-значение)
- Поддержка триггеров вида `триггер: действие параметр=значение`
- Кеширование загруженных данных
- Команда /reload: очистка кеша, перезагрузка всех .txt, событие OnModsReloaded
- Валидация: неизвестные ключи логируются, но не ломают игру

SOLID: парсер, кеш, логгер — отдельные классы, внедряются через интерфейсы.

3. Персонаж и управление

Шаг 3 
Реализуй класс Player на основе IInputProvider.
- Компоненты: HealthComponent, MovementComponent, DodgeComponent, AbilityComponent (3 способности)
- Статы загружаются из персонаж.txt через IModLoader
- Перекат с неуязвимостью, прыжок, приседание
- Анимации через AnimationComponent (заглушки)

Зависимости: IInputProvider, IModLoader, IAssetProvider.
Всё разбито на мелкие компоненты (по Single Responsibility).

4. Оружие (компонентная система)

Шаг 4 
Создай систему оружия на основе композиции:
- Интерфейс IWeaponComponent (FireLogic, Projectile, Trajectory, HitEffect)
- Базовые реализации: SingleShot, Burst, Shotgun (FireLogic); Bullet, Hook, Wave (Projectile); Linear, Sine, Boomerang (Trajectory); Burn, Freeze (HitEffect)
- Сборка оружия из .txt: ствол, снаряд, траектория, эффекты
- Класс WeaponData загружается через IModLoader
- Оружие прикрепляется к игроку, стрельба через IInputProvider

Используй фабрику для создания оружия по конфигу.

5. Враги и ИИ

Шаг 5 
Реализуй систему врагов:
- Базовый класс Enemy с компонентами: HealthComponent, MovementComponent, AIComponent
- AIComponent – интерфейс IEnemyAI с методами UpdateBehavior, методы: Chase, Patrol, Ambush, Flee, Dance и др.
- Загрузка параметров из враг.txt, босс.txt
- Фазы босса, элитность, дроп
- Реакции на попадание/смерть через события

Абстракции позволяют легко добавлять новые типы ИИ.

6. Комнаты и генерация карт

Шаг 6 
Создай DungeonGenerator:
- Чтение карта.txt: тема, этажи, комнаты, секретки
- Генерация сетки комнат, соединение коридорами
- Комнаты: RoomData из комната.txt (тайлы, ловушки, враги, правило)
- Правила комнат реализованы через интерфейс IRoomRule (EchoRule, SilenceRule, MirrorRule, SacrificeRule и т.д.)
- Загрузка через фабрику правил

Поддержка seed и фиксированной раскладки.

7. Сеть (P2P хост-клиент)

Шаг 7 
Интегрируй LiteNetLib (или Mirror) через INetworkService.
- Хост: запуск сервера на указанном порту, загрузка текущих .txt конфигов
- Клиент: подключение по IP:порт
- При подключении клиента хост автоматически отправляет все файлы из папки mods/ и необходимые ассеты (через IAssetProvider)
- Синхронизация: хост авторитарно управляет игровым состоянием, клиенты шлют только ввод
- Команда /reload на хосте вызывает перезагрузку модов и рассылает обновлённые файлы всем клиентам
- Сетевые сообщения строго типизированы, свои классы для каждого типа

8. Мультиплеерные механики

Шаг 8 
Реализуй взаимодействия игроков (кооператив):
- Цепь жизни (LifeLinkComponent)
- Якорь (AnchorComponent)
- Обмен телами (BodySwapComponent)
- Слияние (MergeComponent)
- Жертва, торговля, дуэль
- Все механики работают через интерфейс IMultiplayerInteraction, регистрируются в InteractionManager

Каждая механика – отдельный класс, активируется по команде из .txt или через контекстное меню.

9. Интерфейс, звук, визуал

Шаг 9 
Создай UI-систему, настраиваемую через интерфейс.txt:
- Хп-бар, мана-бар, миникарта, счётчики, инвентарь
- Цифры урона, индикаторы баффов/дебаффов
- Адаптивный UI под разрешения

Звуковая система:
- Интерфейс IAudioService: громкость по категориям, эффекты (эхо, под водой), музыка по ситуациям
- Загрузка настроек из звук.txt

Визуал:
- Пост-обработка, освещение, погода из соответствующих .txt

10. Остальные системы

Шаг 10 
Последовательно добавь:
- Систему предметов (интерфейс IItem, загрузка из предмет.txt)
- Баффы/дебаффы (IBuff) с таймерами, стаканием, заражением
- Крафт (рецепты из крафт.txt)
- Магазин, экономику, квесты, диалоги
- Питомцы, транспорт, мини-игры, секреты, пасхалки

Каждая система реализуется через отдельный интерфейс и сервис, внедряется в GameManager.

11. Редактор модов (Avalonia)

Шаг 11 
Создай отдельное десктопное приложение на Avalonia для визуального редактирования .txt конфигов.
- Вкладки: Персонаж, Оружие, Враг, Комната, Карта, Погода, Предметы, Баффы и т.д.
- Формы с динамической генерацией полей на основе schema.json
- Валидация, автодополнение, предпросмотр спрайтов
- Сохранение в папку mods/
- Приложение не зависит от игры, но может запускать игру для теста

Используй MVVM, все зависимости через DI.

12. Финальная сборка и тесты

Шаг 12 
Настрой мультиплатформенную сборку под Windows, Linux, macOS (Godot).
- Убедись, что пути к файлам используют OS-независимые API
- Протестируй все системы на двух ПК в локальной сети
- Исправь баги, добавь обработку исключений
- Создай README с инструкцией по запуску, созданию модов, подключению через Hamachi/Radmin/белый IP
